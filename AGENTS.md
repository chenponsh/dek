# Agent Memory: Obsidian Vault Organization Rules

本文件记录本 vault 的 AI/Agent 操作约定。后续整理知识库时应优先遵守这些规则；如果来源材料或结构发生变化，应先读取内容并判断语义主题，不要机械套用某一次初始化材料的特殊格式。

## 【通用】约定与原则

#### Markdown 笔记格式规则

- 所有 Markdown 笔记的 YAML frontmatter 与正文之间保留一个空行。

#### 规则维护原则

- 修改本文件或库内规则时，应从全局一致性出发，通读并检查相关约定；新增要求的同时，应删除或改写已经失效、重复或相互冲突的表述，不要只做局部追加。

#### 顶层文件夹约定

- `wiki/`：经过整理、可以发布的正式知识内容。
- `source/`：来源笔记与来源摘录笔记，是从外部来源到正式内容笔记的中间层；与 `wiki/` 同属主要知识库内容，需要纳入 git 版本管理。
- `ingestion/`：来源摄入流程相关的可复用过程记忆与草稿区，需要纳入 git 版本管理。
  - `ingestion/README.md`：来源摄入目录说明。
  - `ingestion/logs/`：来源增量摄入的结构化运行记录，用于下次继续摄入时追踪状态、跳过原因、失败原因和 rough 草稿生成情况。
  - `ingestion/rough/`：摄入后生成、尚未正式整理入 `wiki/` 的 rough 草稿。
- `_raw/`：原始、未经整理的数据来源。通常只作为整理输入，不作为最终知识库内容。
- `_/`：系统工具、临时脚本、处理过程文件等，与知识库正文内容无关。

#### Git 版本管理约定

- `wiki/` 中的正式知识内容与 `source/` 中的来源笔记、来源摘录笔记，均应作为主要版本管理对象并纳入 git。
- `ingestion/` 中的来源摄入说明、摄入记录和 rough 草稿应纳入 git；这是为了让下一次增量摄入能基于上次运行状态继续判断，而不是依赖本地临时缓存。
- `_raw/` 中的原始材料通常不纳入 git。
- `_/` 中与本地环境、临时处理、非通用工具相关的内容通常不纳入 git。
- 原则上不将二进制文件纳入 git；如果确实需要在笔记中使用图片、附件等二进制资源，优先采用图床链接等外部托管方式，只在有明确必要时再纳入版本管理。
- 如果 `_/` 中出现确实需要复用、共享、版本管理的通用脚本或说明，应显式调整 `.gitignore` 例外规则后再纳入 git。

#### 任务执行前的 Git 同步检查

- 每次执行任务前，先检查当前分支、工作区状态和远端更新；至少执行等效于 `git status --short --branch`、`git fetch origin --prune` 的检查，并确认当前分支相对其上游分支是否落后。
- 如果当前分支落后且工作区干净，先使用 `git pull --ff-only` 拉取最新版本，再开始任务；不要使用会改写本地历史或丢弃内容的强制同步方式。
- 如果工作区已有未提交修改，或本地与远端均有提交，不要盲目拉取、自动 stash、reset 或覆盖文件。应先比较本地修改与远端变更涉及的文件和内容：
  - 确认互不重叠且快进拉取安全时，才可使用 `git pull --ff-only`。
  - 存在重叠、分叉或无法确认安全性时，暂停写入并向用户说明潜在冲突，等待用户决定如何处理。
- 只读任务本身不会新增文件冲突，但同步前仍须检查是否存在既有本地修改；不能仅因任务只读就假定拉取一定安全。若不适合改动工作区，可基于已 fetch 的远端版本进行只读核对，并明确说明本地尚未同步。
- 会更新文件的任务应在开始写入前完成上述同步。同步后产生的本地修改通常不会立即冲突；但若任务期间远端再次更新，后续合并或推送前仍可能发生冲突，因此完成任务后应再次检查 `git status`，必要时重新 fetch 并比较远端状态。

#### 来源材料处理原则

1. 不记忆、硬编码某个初始化来源文件的特殊结构。
   - 例如：不要把某次来源材料的编号偏移、目录占位、文件格式等作为永久规则。
   - 新来源材料进入时，应先读取内容、识别目录层级、判断其应归入哪个标签主题。

2. 保留语义顺序和层级。
   - 如果来源有清晰目录结构，优先尊重其目录顺序与层级。
   - 如果来源编号与实际主题层级不一致，应根据内容判断；不确定时再向用户确认。

3. 原始材料只作为输入。
   - 整理完成后，知识应沉淀到 `wiki/`。
   - `_raw/` 中材料可在完成整理和确认后清理。

4. 每个来源笔记可记录自己的整理规则。
   - 若某个来源需要长期增量整理，应在对应来源笔记正文中使用 `## 整理规则` 记录或引用该来源专属的抓取、筛选、命名、存放、正文格式和异常处理方法。
   - 通用规则留在 `AGENTS.md`；某一来源的特殊规则留在该来源笔记内，避免把单一来源的细节固化为全库规则。

#### 演进策略

- 当前先用本文件作为轻量级项目记忆。
- `README.md` 可留作人类读者的项目介绍。
- 如果后续整理流程变得稳定、可复用、需要跨 vault 使用，再考虑抽象为 skill 或脚本化工具。

## 【来源笔记】：循迹获取

适用于 `source/` 下的来源笔记、来源抓取方法笔记和来源摘录笔记。来源笔记是从外部来源到正式 `wiki/` 内容笔记的中间层，不等同于最终知识内容。

#### 来源增量摄入记录

- 每次按用户要求“遍历源笔记”“增量摄入 source”后，应将结构化摄入记录固定保存到 `ingestion/logs/`，不要只放在被 `.gitignore` 忽略的 `_/`。
- 摄入记录命名固定使用：`source_ingest_YYYYMMDD_HHMM_report.json`。`ingest` 表示“将外部来源内容摄入本地知识库”；涵盖抓取远端、解析清洗、主题筛选、写入 source、生成 `ingestion/rough/` 草稿和记录跳过/失败原因。
- 摄入记录 JSON 至少包含：
  - `date`：摄入日期。
  - `report`：逐来源记录，包含 source 笔记路径、摄入状态、远端记录数或最新日期、是否发现符合主题的新增、跳过/失败原因。
  - `rough_created`：本次新增到 `ingestion/rough/` 的草稿笔记列表；无新增时为空列表。
- 摄入状态建议使用稳定枚举值，便于后续脚本或人工复核：
  - `checked_no_new`：已成功核对，未发现符合主题新增。
  - `updated_with_new`：已写入 source 增量，并已生成 `ingestion/rough/` 草稿。
  - `skipped_browser_unavailable`：来源需要浏览器上下文/反爬初始化，但当前环境无可用浏览器。
  - `skipped_no_fetch_rule`：来源笔记标明抓取规则尚未固化，未机械摄入。
  - `failed`：抓取或解析失败，需要人工复核。
- 摄入记录属于可复用的过程记忆，应纳入 git；临时 HTML、接口原始缓存、PDF 下载、调试脚本等仍放入 `_/`，通常不纳入 git。
- 更新 source 笔记 `last_updated` 的边界：
  - 只有成功核对该来源并完成“无新增/已增量写入/已确认无符合主题新增”的判断后，才更新 `last_updated`。
  - 因浏览器不可用、反爬、抓取规则未固化或解析失败而跳过的来源，不更新 `last_updated`，并在摄入记录中写明原因。
- 当前已知摄入依赖/方法：
  - 普通静态列表或 JSON 接口：优先用 Python `requests` 或 PowerShell `Invoke-WebRequest` 抓取。
  - 江苏局 `da.jiangsu.gov.cn` 列表页：普通 `requests` 可能返回 403，加速乐页面；可用 PowerShell `Invoke-WebRequest -UseBasicParsing` 读取 `RawContentStream`。
  - CDE 共性问题：直接 `curl`、`requests` 或普通 HTML 解析通常只能拿到反爬/JS 外壳；需要浏览器页面上下文完成初始化后调用前端 `myAjax("/xxgk/getCommonQuestionList", ...)`，详见 [[source/CDE/CDE_共性问题_SourceFetchMethod.md]]。
  - 国家药典委员会 CPC 专栏：优先使用接口 `https://www.chp.org.cn/three/sys/news/es/list` 和 `https://www.chp.org.cn/three/sys/news/es/detail`，详见对应 source 笔记规则。
  - 上海局常见问题：优先使用 `https://sjcx.yjj.sh.gov.cn/api/v1/qa/list?pageNum=...&pageSize=...`。
- 若摄入时发现本来源方法已稳定复用，应优先沉淀为对应 `_SourceFetchMethod.md`；若只是一次运行的临时脚本，不要放入 `ingestion/logs/`，仍放入 `_/`。

#### 来源笔记结构

- 来源笔记 frontmatter 后一般包含 `## 整理规则` 和 `## 内容` 两部分。
- `## 整理规则` 记录本来源的抓取、筛选、增量、命名、存放、正文格式和异常处理方法；如果已抽取共用方法，则在此处用 Obsidian wikilink 指向共用方法笔记。
- `## 内容` 存放整理后的来源条目；如果来源层级较多，也可以只放摘录目录或分层摘录笔记入口。

#### 共用整理规则笔记

- 当多个来源的整理规则相同或类似时，优先提炼为一个共用整理规则笔记，文件名以 `_SourceFetchMethod.md` 结尾。
- 各来源笔记的 `## 整理规则` 只保留对共用规则笔记的引用，以及确属本来源独有的补充说明。
- 共用规则笔记可在 frontmatter `scope` 中维护适用的来源笔记列表，便于反向追踪。

#### 内容表格格式

- 内容部分一般整理为三列表格：`问题 / 解答 / 发布日期`。
- 若来源本身是问答形式，每个问答拆为一行。
- 若来源不是明显问答形式，以信息标题、文章标题或条目标题作为“问题”，以对应正文或信息内容作为“解答”。
- 表格单元格内换行用 `<br>`；正文中的 `|` 需转义为 `\|`；有效网页链接尽量保留为 Markdown 链接。

#### 文章层级处理

- 有些来源存在“列表页 → 网页文章 → 文章内条目/问答”的信息源层级，网页文章本身是一个需要保留的来源层级。
- 处理文章层级时应根据来源实际情况选择承载方式，不要机械套用某个既有来源的做法：
  - 若单篇文章内容较长、文章数量较多、每篇文章有独立附件/图片/链接，或后续需要按文章增量复核，优先像 `安徽局_许可注册惠企政策超市-药品` 一样为每篇文章建立独立摘录笔记，主来源笔记只保留入口、规则和摘录目录。
  - 若文章层级较轻、文章数量较少、文章内问答可清晰汇总到一个来源笔记表格中，可像 `海南局CDE_服务事项` 一样不拆独立文章笔记，但应在表格中保留所属网页文章标题等可追溯信息。
- 无论采用哪种方式，都应保留“文章标题/发布日期/必要链接或附件”等能回溯原网页文章的信息；如果文章标题不单独作为列，应在对应来源笔记的整理规则中说明其写入位置。

#### 主题筛选

- 本库来源整理以“化学药品制剂”为关键主题。
- 若来源内容混有关键主题以外的信息，只摘取与化学药品制剂及其注册、生产、质量、上市后变更、再注册、说明书标签等直接相关的内容。
- 与关键主题完全无关的内容不摘录，例如仅涉及中药、化妆品、保健品、医疗器械、体外诊断试剂等的内容。
- 若一篇文章或一个列表内混有多主题条目，应在条目/行级别筛选；不要因为文章整体来自“药品”栏目就机械全量摘录。

#### 层级较多的来源

- 当来源内容层级较多、单个来源笔记难以清晰承载时，可建立分层摘录笔记。
- 主来源笔记保留入口、通用规则和摘录目录；每个分层摘录笔记内再按上述表格与筛选规则整理内容。
- 示例：`安徽局_许可注册惠企政策超市-药品` 这类来源可在 `摘录笔记/` 下按文章或子层级拆分。

## 【正式笔记】：分类整理

整理 `_raw/` 或 `source/` 中的来源材料为正式 `wiki/` 内容笔记时，先判断所属标签概念，再建立或复用标签文件夹、标签主页，最后生成内容笔记。

#### 标签、文件夹、标签主页命名规则

1. 每个标签概念使用“编码 + `_` + 标题”的名称。
   - 示例：`01_注册申报`
   - 示例：`0106_受理审查`
   - 示例：`010602_光盘和档案盒封面`

2. 编码规则：
   - 每级顺序号前补 0 至 2 位。
   - 多级编码连续拼接。
   - 编码与标题之间使用半角下划线 `_`。

3. 文件夹结构：
   - 基础目录为 `wiki/`。
   - 每个标签概念对应一个同名文件夹。
   - 子概念文件夹放在父概念文件夹内。
   - 示例：
     ```text
     wiki/
       01_注册申报/
         0106_受理审查/
           010602_光盘和档案盒封面/
     ```

4. 标签路径：
   - 标签路径由各级编码概念名用 `/` 拼接。
   - 标签本身不包含 `wiki/`。
   - 示例：`01_注册申报/0106_受理审查/010602_光盘和档案盒封面`

5. 文件名/标签名清理：
   - 删除标题内部空白，避免标签中出现空格。
   - 标题中 Windows 文件名不允许的字符需要替换为安全字符。
   - `/` 或 `／` 替换为 `、`。
   - 示例：`委托生产/生产许可` → `委托生产、生产许可`。

#### 标签主页 / 标签主笔记规则

1. 每个标签概念文件夹内放一个同名标签主笔记。
   - 示例：`wiki/01_注册申报/01_注册申报.md`
   - 示例：`wiki/01_注册申报/0106_受理审查/0106_受理审查.md`

2. 标签主笔记放入对应标签文件夹内，而不是与文件夹并列放在上级目录。

3. Tag Wrangler 兼容规则：
   - `aliases` 中填写带 `#` 的标签路径。
   - `tags` 中填写不带 `#` 的标签路径。
   - 这样 Tag Wrangler 的标签菜单会显示 `open tag page`，而不是 `create tag page`。

4. 顶级标签主笔记 frontmatter 示例：
   ```yaml
   ---
   aliases:
     - "#01_注册申报"
   tags:
     - "01_注册申报"
   master:
   ---
   ```

5. 子级标签主笔记 frontmatter 示例：
   ```yaml
   ---
   aliases:
     - "#01_注册申报/0106_受理审查/010602_光盘和档案盒封面"
   tags:
     - "01_注册申报/0106_受理审查/010602_光盘和档案盒封面"
   master: "[[wiki/01_注册申报/0106_受理审查/0106_受理审查]]"
   ---
   ```

6. `master` 属性：
   - 指向上级标签主笔记的 Obsidian wikilink。
   - 顶级标签没有上级，保留 `master:` 为空。

7. 标签主笔记正文：
   - 不要添加与文件名重复的一级标题。
   - 如果需要展示子目录或内容笔记，可直接放查询块；查询块前不额外添加“内容笔记”等固定标题。
   - 使用 Dataview 展示内容笔记时，标题链接列使用 `file.link AS 项目`，并放在第一列。

8. 标签主笔记的 Dataview 内容索引表：
   - 标签主笔记可使用 Dataview 为当前标签目录及其子目录内的内容笔记建立索引，避免父级标签页看不到子级内容。
   - 查询块直接放在正文开头，不额外添加“内容笔记”等固定标题。
   - 使用 `TABLE WITHOUT ID` 去掉 Dataview 默认的 `File` 列，再用 `file.link AS 项目` 在第一列显示内容笔记链接。
   - 日期列放在最后，使用 `dateformat(date, "yyyy-MM-dd") AS 日期`，避免显示为本地化的奇怪格式。
   - 示例：
     ```dataview
     TABLE WITHOUT ID
       file.link AS 项目,
       question AS 问题,
       source AS 来源,
       dateformat(date, "yyyy-MM-dd") AS 日期
     FROM "wiki/01_注册申报/0101_注册系统"
     WHERE no != null
     SORT file.folder ASC, no ASC
     ```

9. 标签主笔记的 Dataview 表格样式：
   - 如果 Dataview 表格字号过大、日期换行或显示结果计数干扰阅读，可在标签主笔记 frontmatter 中加入：
     ```yaml
     cssclasses:
       - dv-compact-table
       - dv-overview-table
     ```
   - 这些类名需要配合 `.obsidian/snippets/dataview-compact-table.css` 使用，用于局部压缩表格字号、行距、内边距，控制“项目 / 问题 / 来源 / 日期”四列宽度，保持项目列和日期列不换行，并隐藏 Dataview 表头中的结果计数。
   - 使用 `cssclasses` 的目的是将样式限定在当前标签页，避免影响全库所有 Dataview 表格。

#### 内容笔记构建规则

1. 存放位置：
   - 内容笔记存放在其所属标签概念对应的文件夹内。
   - 示例：`0101_注册系统` 主题下的内容笔记放入 `wiki/01_注册申报/0101_注册系统/`。

2. 笔记标题：
   - 不再使用来源材料中的【问题】内容作为文件名。
   - 文件名使用“标签代码 + `-` + 当前标签目录内序号（补 0 至 4 位）”的组合。
   - 示例：`0101` 标签目录内第 1 条内容笔记命名为 `0101-0001.md`。
   - 标签代码取当前内容笔记所属标签概念名称开头的编码，如 `0101_注册系统` 取 `0101`。

3. frontmatter 字段及顺序：
   ```yaml
   ---
   no:
   date:
   question:
   source:
   tag_pages:
     - 
   tags:
     - 
   ---
   ```
   - `no`：在当前标签目录内，按时间顺序排列的序号。
   - `date`：来源材料中的【发布日期】。当前先使用模糊字段名 `date`，后续如日期含义明确再细分。
   - `question`：来源材料中的【问题】原文。
   - `source`：来源材料中的【解答人】原文。
     - 如果【解答人】为类似 `CDE常见一般性技术问题` 的兜底占位值，表示没有具体信息源，`source:` 留空。
   - `tag_pages`：列表，存放从顶级标签到当前所属标签的所有相应 tag page 的 Obsidian wikilink，按层级从上到下排列。
     - 示例：
       ```yaml
       tag_pages:
         - "[[wiki/01_注册申报/01_注册申报]]"
         - "[[wiki/01_注册申报/0101_注册系统/0101_注册系统]]"
       ```
   - `tags`：列表，存放从顶级标签到当前所属标签的所有相应标签路径，不带 `#`，不包含 `wiki/`，按层级从上到下排列。
     - 示例：
       ```yaml
       tags:
         - "01_注册申报"
         - "01_注册申报/0101_注册系统"
       ```

4. 正文：
   - 正文写入来源材料中的【解答】内容。
   - 不额外重复写问题标题，除非用户另有要求。

5. 链接处理：
   - 正文中如果出现有效网站链接，应维护为 Markdown 链接。
   - 如果原文是“说明文字 + URL”，不要把整句变成链接；应尽量保留说明文字为普通文本，并将 URL 单独一行写成链接。
   - 示例：
     ```markdown
     1. 关于某事项的通知

        [https://example.com/page](https://example.com/page)
     ```

6. 内容改写边界：
   - 不改变来源材料的实质含义。
   - 可以进行轻量 Markdown 排版，如将明显的编号清单整理为有序列表。
   - 不确定是否属于格式整理还是内容改写时，先保守保留原文结构。
